### Decisions / notes (M5)

- **Tab counts = backlog totals, not post-filter counts.** The count tells you how many cards
  lack the date; the filter is a temporary lens over the visible list. Documented.
- **Quarter/year cells are compact** (a count badge, not chips) — full chips per day across 3–12
  months would overflow. Click-through is via the week/month ranges.
- **Obsidian owns an open `.base` file** and rewrites it (normalizing/unquoting YAML),
  clobbering external edits to view config. To change view options reliably from the CLI:
  detach all bases leaves first, then edit the file, then open. (Added to the harness lore.)
- **Pre-existing noise:** Obsidian logs `ENOENT … Archive/2026/Task F.md` — a stale workspace
  reference from the M4b archive round-trip (file moved back). Not from plugin code.

### Follow-up: panel chips still clipped — flexbox shrink bug

The wrap fix alone wasn't enough: with the panel holding 941 chips, each `.kap-cal-card` (a flex
item in the scrolling `.kap-panel-list` flex column) was shrunk by the default `flex-shrink: 1`
to ~11.5px — below its 15.9px line height — and `overflow: hidden` (from `truncate`) clipped the
text vertically. Live `getComputedStyle` confirmed: `height 11.5px` vs `line-height 15.9px`.
Fix: add `shrink-0` to the base `.kap-cal-card` so chips keep their natural height and the list
scrolls instead. Re-measured live: single-line chips 27px, wrapped 43px, no clipping.

### Follow-up: panel chips must be uniformly sized

Wrapping made chip heights vary (27px single-line vs 43px wrapped), which looked inconsistent.
Per the consistency requirement, panel chips became a **uniform fixed height** single line —
then (next) a two-line clamp once two lines were requested.

### Follow-up: active-period highlight (and Board/Calendar + panel tabs)

The active time-period button never highlighted — and neither did the Board/Calendar mode switch
or the Unplanned/No-deadline tabs. Root cause (confirmed live): our buttons live in
`@layer kap-components`, but Obsidian styles `<button>` with **unlayered** rules, and unlayered
declarations beat _any_ layered rule regardless of specificity. The active background resolved to
`--interactive-normal` (Obsidian's default), not our `--interactive-accent` — even the
`.kap-root`-prefixed (0,2,0) selector lost. Fix: `!important` on the three `-active` rules
(`kap-mode-btn-active`, `kap-panel-tab-active`, `kap-range-btn-active`); layered `!important`
_does_ outrank unlayered normal declarations. Each carries a `stylelint-disable
declaration-no-important -- …` block explaining why. Live-verified: all three active states now
render the accent (`rgb(91,163,230)`).
